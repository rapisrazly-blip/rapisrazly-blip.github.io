# cc-lp-v2 — AI LP制作ツール

対話だけでプロ品質のランディングページを制作するツール。
HTML/CSSで完成させた後、Figma互換のHTMLにビルドしてからインポートし、編集可能なデザインに変換する。

> **ユーザー像**
> このツールを使うのは、**Claude Code を使ってLPを作りたい第三者クライアント**（プログラミング未経験者）。
> ユーザーはチャットで話しかけるだけで、ファイル生成・コマンド実行はすべて **AI（あなた）が代行** する。
> ターミナル操作・コード編集はユーザーに依頼しない。

> **v2 の変更点（v1からの差分）**
> - Phosphor Icons を Web Components から **インラインSVG** に変更（html.to.design 互換）
> - Figma送り用の **ビルドスクリプト** `tools/build-figma-ready.mjs` を追加（CDN画像 & ローカル画像を自動でBase64化）
> - Phase 6 に「Figmaビルド」ステップを追加
> - **ヒアリング時に AI が参考 LP を Web 検索で自動提案**（ユーザーが参考サイトを思いつかなくても進められる）
> - **画像素材の自動補完**: ユーザーが画像を用意しなくても AI が自動で入れる（ロゴは SVG 生成、顔写真は pravatar、ヒーロー/商品写真は Unsplash から AI が検索して選定）
> - **レスポンシブ対応が必須に**: PC（> 768px）+ モバイル（≤ 768px）の2パターン対応（768pxブレークポイント1つでシンプルに切替）
> - **参考LPディープ分析の必須化**: Phase 2 で参考LPを `WebFetch` で取得→`lp/.analysis.md` に構造化して保存→Phase 3〜5で常時参照（AIぽさの主要原因「参考LPを選ぶだけで活用しない」を解消）
> - **情報密度ガードレール**: セクション別の必須要素数を明文化（FVは5要素、特徴は4ベネフィット以上、お客様の声は3件以上など）。情報不足は業界標準の仮案で埋めて「スカスカ」を根絶
> - **AIぽさ自己レビュー**: Phase 5 の品質チェックを拡張し、参考LP継承度/情報密度/視覚リズム/AIぽさの兆候を AI が自分で検出→自動修正してからユーザーに提示

## トリガー

ユーザーが以下のような発話をしたら、このファイルの手順に従って実行する:
- 「LPを作りたい」「ランディングページを作りたい」「ホームページを作りたい」「Webサイトを作りたい」
- LP・ランディングページ・ホームページの制作を意図する発話全般

## 共通ルール

- ユーザーはプログラミング未経験者。コードの説明は不要
- **コマンド実行は AI が代行する**（ユーザーにターミナル操作を指示しない）
- **ユーザーから画像ファイルが渡されたら `lp/assets/` に保存する**（ファイル名は半角英数字に整える）
- 修正は何度でも対応する
- 毎回違うデザインにする。同じスタイルを使い回さない
- **制作開始前に必ず `guidelines/design-system.md` を読むこと**

```

## LP制作スキル

### 全体フロー

```
Phase 1: ヒアリング（ビジネス情報 + デザインの方向性）
Phase 2: ビジュアルテーゼの決定（デザインの方向性を1文で定義）
Phase 3: LP構成・コピー設計（セクション構成とテキストを設計）
Phase 4: HTML/CSS生成（ローカルでプレビュー可能なLP）
Phase 5: HTML上で修正ループ（完成度を上げる）
Phase 6: Figmaにインポート（html.to.design プラグイン）
```

**原則: HTMLの段階で完成度を最大限に上げてから、Figmaにインポートする。**

---

### Phase 1: ヒアリング

ユーザーに以下を順番に聞く。一度に全部聞かず、会話形式で2〜3問ずつ聞くこと。

#### ヒアリングの流れ（推奨）
1. ビジネス情報（Q1〜5）を会話形式で聞く
2. ビジネス情報が揃ったら、**AI が参考 LP を Web で検索して3つ提案**（Q6）
3. ブランドカラー（Q7）と CTA リンク先（Q8）を確認
4. ロゴの有無を軽く確認（Q9）+ コンテンツ情報（Q10〜12）は任意
5. **詳細な画像素材は Phase 1 では聞かない**。Phase 3 で構成確定後に「セクション別に何が必要か」をまとめて案内する

#### ビジネス情報（必須）
1. **ビジネスの内容**: 何を売っている／提供しているか
2. **ターゲット**: 誰に向けたサービスか
3. **LPの目的**: 見た人にどんな行動を取ってほしいか（申し込み、問い合わせ、LINE登録、資料請求等）
4. **強み・差別化**: 競合と比べて何が違うか（3つ程度。なければAIが提案する）
5. **実績**: 数字で示せるもの（なければ省略OK）

#### デザインの方向性（必須）
6. **参考サイト**: **AI が能動的に Web 検索で候補を3つ提案する**:
   - 探し方: ビジネス内容・ターゲット・業界・雰囲気を踏まえ、Web検索（`WebSearch`）で良質な日本語LPまたは海外LPを探し、必要に応じて `WebFetch` で内容を確認する
   - 探し場所の例: Land-book / SaaSLandingPage / Awwwards / 各業界の有名LP / 「[業界名] ランディングページ 参考」で検索
   - 提案内容: 各候補に以下を添える
     - LPのURL
     - 一言で雰囲気（例: 「ミニマル・ホワイト基調・大きな写真主体」）
     - **なぜこのLPを参考にすべきか**（ターゲットとの相性、使える装飾・構成の特徴など）を1〜2行
   - ユーザーの動き: 気に入ったものを選ぶ（1〜複数可）、または自分でURLを追加
   - **最低1つは参考サイトを確定してから Phase 2 へ進む**（これがデザイン品質を決定的に左右する）
   - ユーザーが最初から「このサイトがいい」とURLを出してきた場合は検索をスキップし、そのまま採用する（+ 必要に応じて AI が補足候補を出す）
7. **ブランドカラー**: あれば（なければビジネスに合わせてAIが選定）

#### CTAリンク先（必須）
8. **CTAボタンのリンク先URL**: 申し込みフォーム、LINE登録URL、問い合わせページ等。**必ず聞くこと。** リンク先が決まっていなければ `#` で仮置きし、後から差し替える旨を伝える

#### ビジュアル素材の予告（軽く確認）
9. **ロゴ画像**: 会社・サービスのロゴがあれば、チャットに添付して渡してもらう（ヘッダー・フッターで使用）

> **ロゴ以外の画像リクエストはこの段階ではしない。** Phase 3 で LP 構成が確定した後、
> セクション別に「どんな画像があると効果的か」を具体的にチェックリスト形式で提示して、まとめて聞く。
>
> ロゴが添付された場合は AI は `lp/assets/` 配下に保存する（ファイル名は半角英数字に整える。例: `logo.png`）。
>
> ユーザーには「他に使えそうな画像があれば手元で整理しておいてください。後で具体的にリクエストします」と予告しておく。

#### コンテンツ情報（任意）
10. **お客様の声**: あれば1〜3件
11. **料金**: 載せるか、載せないか
12. **既存の資料**: 提案書やサービス概要があれば共有してもらう（内容を反映できる）

### Phase 2: 参考LPディープ分析 + ビジュアルテーゼの決定

**このフェーズの狙い**: 参考LPを「知ったかぶり」でなく、具体的な要素（レイアウト/情報密度/装飾/ビジュアル）として抽出し、生成時のレシピに変換する。
**ここを手を抜くと、AI がスカスカでAIぽいLPを吐く直接的な原因になる。**

---

#### ステップ1: 参考LPのディープ分析（必須）

**必ず `WebFetch` で参考LPを実際に取得して分析すること。「雰囲気でなんとなく再現」は禁止。**

##### アクション
1. Phase 1 の Q6 で確定した参考LPのURLを `WebFetch` で取得
2. 複数あれば全てを取得
3. 取得できたら下記の **分析書テンプレート** に沿って具体的要素を抽出
4. 抽出結果を `lp/.analysis.md` として保存（AI がファイルとして書く）
5. 以降の Phase 3・4・5 で必ずこのファイルを参照する

##### WebFetch が失敗した場合の fallback
以下のいずれかに遭遇したら、ユーザーにスクショ提供を依頼する:
- 動的レンダリング（JS必須）でHTMLが空
- Cloudflare などでボット弾き
- 403/503/404 エラー

ユーザーに伝える文言:
```
参考LPの取得に失敗しました（動的レンダリング or ボット保護のため）。
以下のスクショをチャットに貼っていただけますか？
- ファーストビュー全体（1枚）
- 主要セクション2〜3枚（特徴/実績/料金あたり）
- フッター手前の最終CTA周辺（1枚）
```
スクショから同じ分析書項目を埋める。

##### 分析書テンプレート（`lp/.analysis.md` に保存）

```markdown
# 参考LP分析書

## 基本情報
- URL: [参考LP URL]
- 一言で: [例: ミニマル・白基調・巨大写真主体]

## セクション構成（順番通り）
1. [セクション名] — [役割・情報量の濃さ]
2. ...

## FVの組み方（最重要）
- レイアウト: [2カラム / 1カラム中央 / フルブリード / 背景動画]
- テキスト位置: [左/右/中央] / テキスト幅: [画面の何%]
- ビジュアル種別: [写真 / ダッシュボードモックアップ / 製品ショット / イラスト / 動画サムネ / アニメーション]
- CTA: [個数 / 文言の傾向 / 副CTA（「詳細を見る」等）の有無]
- 信頼要素: [ロゴバー(何社) / 実績数字 / 受賞アイコン / 評価スター / ユーザー数]
- 見出しのサイズ感: [例: 画面の1/4を占める超大見出し、clamp(60px, 8vw, 120px)クラス]

## カラーパレット
- メインカラー: [#xxxxxx]
- アクセント: [#xxxxxx]
- 背景トーン: [明基調 / 暗基調 / グラデ / セクション毎に明暗交互]
- セクション背景バリエーション: [何パターン使われているか]

## タイポグラフィ
- 見出し書体: [Serif / Sans-serif / 手書き系]
- 本文書体: [Serif / Sans-serif]
- 見出しと本文の対比: [太さ/サイズ/書体の対比が効いているか]
- 超大見出しの有無: [あり/なし、最大サイズ感]

## 装飾要素
- 背景: [単色 / 薄グラデ / パターン / 画像バック / セクション毎に変化]
- オーナメント: [セクション番号 / 見出しライン / グラデ円 / ドットパターン / 矢印 / リング]
- アニメ: [ホバー / スクロールで出現 / パララックス / なし]

## ビジュアルアセット(重要)
- 種別: [スクショ / モックアップ / イラスト / 写真 / 数字表現 / グラフ / チャート / アイコン]
- 使い方: [例: FVに巨大なダッシュボードモックアップ1枚、特徴セクションに小さなスクショ3枚、実績に超大きい数字]

## 情報密度
- 1セクションあたりの要素数: [例: FVに主見出し+サブ+CTA2個+ロゴバー+信頼数字=5要素、特徴セクションに4カラム×2行]
- サブコピーの長さ: [1行短め / 2〜3行詳しめ / 箇条書き型]
- 詰め込み度: [余白多め / バランス / 詰め込み型]

## このLPの「強み」TOP5（コピーすべき要素）
1. [例: FVの左右2カラムで左に巨大見出し、右に製品モックアップを置くバランスが完璧]
2. ...
3. ...
4. ...
5. ...

## このLPから継承する具体ルール
- FVは〜する
- セクション背景は〜する
- 装飾は〜を使う
- タイポは〜の対比を効かせる
- ビジュアルアセットは〜を入れる
```

##### 複数の参考LPがある場合
- 全ての分析書を埋める
- 最後に「共通する軸」「使い分ける軸」を末尾に追記して、どのLPのどの要素を採用するか決める

##### フォールバック（参考サイトなしで例外的に進める場合）
ユーザーが「参考サイトは不要、AI に任せる」と明言したケースのみ、`guidelines/design-system.md` のセクション1「美的方向性の決め方」のトーン一覧から、ビジネスに最適なものを選び、分析書のテンプレートをAIが想像で埋める。**この場合は品質が下がることをユーザーに明示する。**

#### ステップ2: ビジュアルテーゼの確定

分析書を元に、デザインの方向性を1文で定義し、具体パラメータをまとめる。

##### ユーザーに提示する内容
```
【ビジュアルテーゼ】
「深いネイビーと白のコントラストに、シアンブルーのアクセントが映える、
抑制された高級感のあるコーポレートLP」

【選定理由】
- 法人向けサービスなので信頼感・コーポレート感を重視
- ネイビー×シアンで先進性を表現
- 余白を多めに取って高級感を出す

【フォント】Noto Sans JP (900 / 400)
【カラー】メイン #0a1628 / アクセント #0ea5e9 / 背景 #f8fafc

【参考LPから継承する具体ルール】(ここが重要)
- FVは参考LP X のように左にテキスト60%幅・右に巨大なダッシュボードモックアップ
- 見出しは clamp(60px, 8vw, 110px) クラスの超大見出しで視認性圧倒
- セクション背景は白→薄グレー→白→ネイビー暗基調 の4パターンで交互変化
- 特徴セクションは参考LP Y のように4カラム×2行=8ベネフィット詰め込み
- 実績は特大数字 + 比較で見せる（参考LP Z 方式）
- 装飾は左上にグラデ円、見出しの上に番号ライン
```

ユーザーの「OK」を得てから次に進む。
**継承ルールは Phase 3・4 で必ず参照する**ため、曖昧な記述でなく具体パラメータで書く。

### Phase 3: LP構成・コピー設計

> **このフェーズの必須参照先**:
> - `lp/.analysis.md`（Phase 2 で作成した参考LP分析書）
> - `guidelines/design-system.md` セクション2「情報密度ガードレール」
>
> **構成・コピーを作る前に必ずこの2つを読み、**
> 1. 参考LPのセクション構成・情報量を踏襲する
> 2. 情報密度ガードレールの必須要素数を満たす
>
> この2つを外すと、生成物がスカスカ・AIぽくなる。

#### 鉄板LP構成（8セクション）

| # | セクション | 目的 | ビジュアル要素 |
|---|-----------|------|-------------|
| 1 | ファーストビュー | 3秒で価値を伝える | キャッチ + CTA + イラスト or CSS装飾 |
| 2 | 課題提起 | 共感を得る | 悩みリスト + unDrawイラスト |
| 3 | 解決策/比較 | 差別化を示す | 比較テーブル or 特徴カード + アイコン |
| 4 | サービス詳細 | 具体的に理解させる | フローステップ + アイコン |
| 5 | 実績/成果 | 信頼を得る | 数字実績 + unDrawイラスト（成長系） |
| 6 | お客様の声 | 社会的証明 | プレースホルダー顔写真 + コメント |
| 7 | 料金 | 行動ハードルを下げる | 料金カード（featured強調） |
| 8 | 最終CTA + FAQ | 行動させる | CTA + unDrawイラスト（FAQ横） |

ビジネスに合わせて追加・削除・順番変更はOK。
**参考LP分析書に「このセクションがある」と書かれているものは優先的に追加する。**

#### 情報密度の担保（必須）

`guidelines/design-system.md` セクション2「情報密度ガードレール」のテーブルに沿って、**各セクションの必須要素を全て満たすように構成を設計する**。

**主要な注意点**（ガードレールから抜粋）:
- FV: 主見出し + サブ + **CTA 2個以上** + ビジュアル + 信頼要素（ロゴバー/実績数字）の **5要素必須**
- 特徴: **4ベネフィット以上**（2×2 or 3×2グリッドで詰める）
- 実績: 具体的な数字 **3つ以上** を超大サイズで
- お客様の声: **3件以上**
- 比較表: **5項目以上 × 2〜3サービス**
- 料金: **2〜3プラン** × 各プランに **5特徴以上**
- FAQ: **5問以上**

**ヒアリング情報で埋まらない項目の扱い**:
- 無理に削除せず、AI が **業界標準の仮案** を生成して仮入れする
- ユーザーへ: 「ここはAIが仮で入れたので、後で実データに差し替えてください」と明示
- 「情報が足りないから省略」は禁止（スカスカの原因）

#### コピー作成のルール
- キャッチコピーは**20文字以内**
- サブコピーは**1〜2行**
- 各セクション見出しに**具体的な数字や結果**を入れる
- CTAは**動詞形**:「今すぐ申し込む」「無料で相談する」
- 30%削っても意味が通るなら削り続ける
- 専門用語は避け、ターゲットの言葉で書く

#### ステップ1: ユーザーに構成・コピーを提示
- 全セクションの見出し + 本文テキスト + CTA文言
- Markdown形式で提示
- 参考LP分析書で「継承する」と決めたレイアウト特徴（FVの組み方など）も一言添えて、ユーザーに合意を取る
- **AI が仮で埋めた項目（仮のお客様の声・想定数字・仮FAQ等）は明示的にマーク**して、ユーザーに「これは仮案です」と伝える
- 「OK」を得てから **ステップ2（画像素材の具体リクエスト）** に進む

#### ステップ2: セクション別に画像素材をリクエスト（構成確定後）

ユーザーから構成・コピーの「OK」が出たら、HTML 生成（Phase 4）に入る前に、
**確定した構成にフィットする画像を、セクション別にチェックリスト形式でリクエストする。**

これは Phase 1 の Q9（ロゴ確認）の続きに当たる。Phase 1 では構成が見えていなかったため
具体的な画像リクエストは保留していた。ここで初めて「どんな画像があると効果的か」が明確になる。

##### ユーザーに提示するメッセージの型

確定した構成を元に、セクションごとに必要になりそうな画像を具体的に列挙する。**各項目には「添付しなかった場合はどう自動補完されるか」を明記する**。以下は例（実際は構成に合わせて調整する）:

```
LPの構成とコピーが確定しました！
HTML 生成に入る前に、手元にある画像があれば添付してください。

全部そろえる必要はありません。
**添付されなかった部分は AI が自動でいい感じの画像を入れます。**
気に入らなければ後からいつでも差し替えできます。

□ 1. ロゴ画像（ヘッダー・フッター用）
    → 添付なしの場合: サービス名の頭文字を使ったシンプルロゴをSVGで自動生成
□ 2. ファーストビューのメインビジュアル
    → 添付なしの場合: Unsplashからビジネスに合う写真をAIが選んで配置
□ 3. 「解決策」セクションで見せたいスクショ or 図
    → 添付なしの場合: Unsplashの関連写真 or unDrawイラスト
□ 4. 「サービス詳細」セクションのフロー図 or 管理画面スクショ
    → 添付なしの場合: Unsplashのビジネス写真 or unDrawイラスト
□ 5. 「実績」セクションの数字グラフ or 受賞歴スクショ
    → 添付なしの場合: unDrawの成長系イラスト
□ 6. 「お客様の声」用の顔写真（顔出しNGならダミーでOK）
    → 添付なしの場合: pravatar.cc のランダム顔写真

添付したいものだけチャットに貼ってください。
「このまま進めて」で全部自動補完して先に進みます。
```

##### AIの動き
- ユーザーが画像を添付してきたら、`lp/assets/` に保存する
- ファイル名は半角英数字に整え、どのセクションで使うかわかる名前にする
  - 例: `logo.png`, `hero-main.jpg`, `feature-dashboard.png`, `testimonial-01.jpg`
- スキップされた/添付されなかった箇所については、Phase 4 で **後述の「画像の自動補完ルール」に従って AI が画像を選定・生成する**
- 添付/スキップが確定したら、「以下の画像を使ってHTMLを生成します。進めていいですか？」と確認（スキップ分は自動補完すると明言する）
- 「OK」「進めて」で Phase 4 へ

### Phase 4: HTML/CSS 生成

承認された構成・コピー・ビジュアルテーゼを元に `lp/index.html` を生成する。

> **このフェーズの必須参照先（生成前に再読する）**:
> 1. `lp/.analysis.md` — 参考LP分析書。「継承する具体ルール」を HTML に落とし込む
> 2. `guidelines/design-system.md` — デザインシステム全般。特に以下を厳守:
>    - セクション2「情報密度ガードレール」（必須要素数）
>    - セクション4「タイポグラフィ」のサイズと対比ルール
>    - セクション6「レスポンシブ（必須）」
>    - セクション9「CSS装飾の必須運用ルール」

#### 必ず `guidelines/design-system.md` を参照して以下を守る:
- 美的方向性 → ビジュアルテーゼに従う
- 構成の原則 → FVは主要5要素 + 情報密度ガードレールに従う
- **情報密度** → 各セクションの必須要素数を満たす（スカスカ禁止）
- コピーの原則 → CTA3箇所以上、動詞形
- タイポグラフィ → 選定したフォントペアリング + FV超大見出し `clamp(48px, 7vw, 110px)` 以上
- カラーパレット → 選定した配色をCSS変数で定義 + セクション背景3トーン以上で交互変化
- レイアウト → 8pxグリッド、カードデフォルト禁止
- アイコン → Phosphor Icons（インラインSVG）
- イラスト → unDraw CDN（1LPに3〜5個、明るい背景セクションに配置）
- CSS装飾 → **全セクションに装飾要素を最低1つ**（グラデ円/セクション番号/見出しライン/ドットパターン/リング 等を3種類以上で使い分け）
- モーション → **許可されるのはホバー効果とヒーロー入場（CSS `@keyframes` で自動再生）のみ**。**スクロール出現系（`.fade-in { opacity: 0 }` → JSで `.visible` 付与、`data-aos`、ScrollReveal、Intersection Observer系）は禁止**。Figma取り込み時にJSが動かず要素が消える/ズレるため（詳細は `guidelines/design-system.md` セクション10）

#### 技術ルール
- **1ファイル完結**: HTML + CSS（`<style>`タグ内）で1つの `index.html`
- **アイコン**: Phosphor Icons は **インラインSVG** で埋め込む（https://phosphoricons.com/ でSVGコードをコピーしてHTMLに直接貼り付け）。Web Components（`<ph-icon>`）は使わない
- **画像の扱い**:
  - **ユーザー提供画像**（ロゴ・商品写真等）: `lp/assets/` に保存し、`<img src="./assets/[ファイル名]">` で参照する
  - **unDraw SVG**（装飾イラスト）: `<img src="https://cdn.jsdelivr.net/npm/undraw-svg@1.0.0/svgs/[名前].svg">` でCDN参照でOK
  - **Unsplash写真**（自動補完のビジネス写真）: `<img src="https://images.unsplash.com/photo-xxxxx?w=1600&auto=format&fit=crop&q=80">` でCDN参照でOK
  - **pravatar顔写真**（自動補完の顔写真）: `<img src="https://i.pravatar.cc/150?u=[seed]">` でCDN参照でOK
  - 作業中はどれもそのままで良い。Phase 6 のFigmaビルドステップで **ローカル画像・CDN画像（Unsplash/pravatar/unDraw全て）が自動的にBase64化** される
- **外部依存は以下のみ許可**:
  - Google Fonts（`<link>` タグ）
  - unDraw SVG / Unsplash 画像 / pravatar 顔写真（上記の通り）
- **レスポンシブ対応（必須）**: PC（> 768px, 1280px幅想定）+ モバイル（≤ 768px）の2パターン
  - ブレークポイントは **768px の1つのみ**
  - すべてのグリッドはモバイルで1列に折り返す（`grid-template-columns: 1fr`）
  - FVのテキスト+ビジュアル2カラムはモバイルで縦並びに
  - CTAボタンはモバイルでフル幅（`width: 100%`, `min-height: 48px`）
  - 画像は必ず `max-width: 100%; height: auto;`
  - 詳細ルールは `guidelines/design-system.md` セクション6「レスポンシブ（必須）」を参照
- **アニメーション（Figma互換のため制約あり）**:
  - **許可**: (a) ヒーロー入場（CSS `@keyframes` + `animation: ... both;` で自動再生）、(b) ホバー効果（`:hover` + `transition`）
  - **禁止**: `opacity: 0` や `transform: translateY()` で初期状態を隠し、JS／`.visible` クラスで見せるタイプのスクロール出現系アニメーション全般
  - 禁止の理由: Figma（`html.to.design`）はJSを実行しないため、初期状態のまま透明・ズレた位置で取り込まれてしまう
  - 詳細ルールと実装例は `guidelines/design-system.md` セクション10「モーション」を参照
  - 念のため `npm run build:figma` 時に Figma取り込み用リセットCSS が自動注入されるが、そもそも禁止パターンを書かないのが原則
- **JSフレームワーク不要**: バニラHTML/CSSで完結（Intersection Observer を使った出現アニメーションも書かない）

#### 画像の自動補完ルール（Phase 3 ステップ2でスキップされた画像の補完）

ユーザーが添付しなかった画像は、以下のルールで **AI が自動補完する**。

##### ① ロゴ（スキップ時）

サービス名の頭文字を使ったシンプルロゴをインラインSVGで生成する。

- 英語名なら **頭文字1〜2字**、日本語名なら **頭の漢字1字 or 英語略称の1〜2字**
- ブランドカラー（Phase 2で定義したメインカラー）で塗る
- 形: 円 / 角丸四角 / 斜体の大文字 など、ビジュアルテーゼのトーンに合わせる

```html
<!-- 例: サービス名 "Acme" の場合 -->
<svg width="40" height="40" viewBox="0 0 40 40">
  <circle cx="20" cy="20" r="20" fill="var(--brand-primary)"/>
  <text x="20" y="27" text-anchor="middle" fill="white" font-size="18" font-weight="900" font-family="sans-serif">A</text>
</svg>
```

ヘッダーとフッターに同じSVGを使う（複製）。

##### ② 顔写真（お客様の声セクション）

`pravatar.cc` でランダムな顔写真を取得する。

```html
<img src="https://i.pravatar.cc/150?u=customer1" alt="お客様の声 1">
<img src="https://i.pravatar.cc/150?u=customer2" alt="お客様の声 2">
<img src="https://i.pravatar.cc/150?u=customer3" alt="お客様の声 3">
```

- `?u=` のseed は `customer1`, `customer2` のように一意にする（同じseedだと同じ顔が出る）
- 日本人顔にしたい場合: `https://i.pravatar.cc/150?img=1` 〜 `img=70` の番号で指定可能（事前に複数試して合うのを選ぶ）

##### ③ ヒーロー・商品・ビジネス写真（スキップ時）— Unsplashから自動検索

AI が Phase 2 のビジュアルテーゼと Phase 1 のビジネス内容を踏まえ、**Unsplash から適切な写真を検索して直接URLを組み込む**。

**検索手順:**
1. セクションごとに2〜3個のキーワードを生成する
   - 例: BtoB SaaS のファーストビュー → "business meeting professional", "office laptop team"
   - 例: パーソナルトレーニングのFV → "gym workout motivation", "fitness trainer"
   - 例: カフェのFV → "cafe interior cozy", "coffee latte art"
2. `WebFetch` で Unsplash 検索ページを取得:
   - URL形式: `https://unsplash.com/s/photos/[キーワード（ハイフン区切り）]`
   - 例: `https://unsplash.com/s/photos/business-meeting`
3. 返ってきた HTML から `images.unsplash.com/photo-[ID]` 形式の画像URLを抽出（最初の1〜3枚から選ぶ）
4. 選んだURLに幅とフォーマット指定のクエリを付けて HTML に組み込む:

```html
<img src="https://images.unsplash.com/photo-[ID]?w=1600&auto=format&fit=crop&q=80" alt="[適切なalt]">
```

- `w=1600`（幅1600px）はファーストビュー用。セクション内の小さい画像は `w=800` 程度に調整
- 複数箇所に使う場合は **セクションごとに違うキーワードで検索する**（同じ写真を使い回さない）
- 写真のトーンはビジュアルテーゼに合わせる（ミニマル系なら余白多い写真、エネルギッシュ系なら動きある写真）

##### ④ 装飾イラスト（スキップ時）

既存のルール通り、unDraw CDN を使う。

```html
<img src="https://cdn.jsdelivr.net/npm/undraw-svg@1.0.0/svgs/[名前].svg" alt="[説明]">
```

#### 自動品質チェック（生成直後に実行）

**まず最初に必ず「AIぽさ自己レビュー」を実行し、問題があれば自動修正してからユーザーに見せる。** 生成しっぱなしで渡すのは禁止。

##### ① AIぽさ自己レビュー（最重要・必須）

**観点**: 参考LPのクオリティに近づいているか、スカスカでAIぽくないか。以下を全て自問すること。

**A. 参考LPの継承度チェック**
- `lp/.analysis.md` を開き直して、「このLPから継承する具体ルール」の各項目が HTML に実装されているか
- 特にFV の組み方（レイアウト/ビジュアル種別/CTAの数/信頼要素）が分析書通りになっているか
- ビジュアルアセットの種類（モックアップ/スクショ/数字/グラフ）が分析書で指定したものと合っているか

**B. 情報密度チェック**
- `guidelines/design-system.md` セクション2「情報密度ガードレール」のテーブル通りに、**全セクションの必須要素数を満たしているか**
- FV: 主見出し + サブ + CTA **2個以上** + ビジュアル + 信頼要素の **5要素がすべて揃っているか**
- 特徴セクション: **4ベネフィット以上**あるか
- 実績: 数字 **3つ以上**あるか
- お客様の声: **3件以上**あるか
- 比較表: **5項目以上 × 2〜3サービス**か
- 料金: **2〜3プラン × 5特徴以上/プラン**か
- FAQ: **5問以上**か
- ヒアリング情報不足で埋まっていないセクションがあれば、AI が業界標準の仮案で埋める（ユーザーに後で確認する旨を伝える）

**C. 視覚的リズムチェック**
- セクション背景色が **3トーン以上** で交互変化しているか（全セクション白基調は NG）
- 各セクションに **装飾要素が最低1つ** あるか（グラデ円/ドットパターン/セクション番号/見出しライン/矢印）
- **タイポの対比**が効いているか: H1 ≥ `clamp(48px, 6vw, 96px)`、注記 14px などの極端な差
- FV には **超大見出し**（`clamp(48px, 7vw, 110px)` 以上、`font-weight: 900`）が入っているか

**D. AIぽさの兆候チェック（見つけたら直す）**
- [ ] セクションが「見出し + 3要素のカード横並び」ばかりの繰り返しになっていないか → レイアウトバリエーションを増やす
- [ ] 余白が均等で単調になっていないか → 一部のセクションで詰め込み型にする
- [ ] 装飾が全くないセクションがないか → 各セクションに1つは装飾を入れる
- [ ] 色のトーンが均質（パステル一色、ダーク一色など）になっていないか → セクション毎にトーン変化
- [ ] 「〜を最大化」「〜を実現」「〜で効率化」のような抽象語まみれになっていないか → 具体的な数字や固有名詞に置き換える
- [ ] CTAが「お問い合わせ」だけになっていないか → 「無料で資料をもらう」「7日間無料で試す」など動詞+具体性に変える

**E. FV 専用チェック**
- FV に **ビジュアルアンカー（目を引く巨大要素）** があるか（製品モックアップ/巨大な数字/特大写真/強い背景グラデ等）
- FV の見出しが **画面幅の30〜50%** を占めるほど大きいか（小さすぎるとAIぽくなる）
- FV の空きスペースが不自然に広くないか（参考LPは情報で埋まっているはず）

→ **A〜E のいずれかで問題があれば、AI が自動で `lp/index.html` を修正してから次のチェックに進む。**

---

##### ② 従来の品質チェック

**Litmusチェック**
- ブランド/サービス名がファーストビューで最も目立っているか
- ビジュアルアンカー（目を引く要素）がファーストビューにあるか
- 見出しだけ読んでLPの内容が理解できるか
- 各セクションが1つの仕事だけしているか
- CTAが3箇所以上あるか
- コピーをさらに削れないか

**ビジュアルチェック**
- unDrawイラストが暗い背景で見えにくくなっていないか
- セクションの背景色が交互のリズムになっているか
- ホバーアニメーションが全カード・ボタンに付いているか
- CTAボタンに正しいリンク先URLが設定されているか

**実装チェック**
- 画像に `alt` がある
- `transition: all` を使っていない
- リンクは `<a>` タグを使っている

**Figma互換チェック（アニメーション）**
- [ ] `opacity: 0` で初期状態を隠して JS で見せる実装になっていないか（grep で `opacity: 0` / `opacity:0` を検索）
- [ ] `.fade-in`, `.fade-up`, `.slide-up`, `.reveal`, `.scroll-animate`, `.animate-on-scroll`, `[data-aos]` などスクロール出現系のクラス・属性を使っていないか
- [ ] `IntersectionObserver`, `getBoundingClientRect`, AOS.js, ScrollReveal などの JS スクロール発動コードが混入していないか
- [ ] ヒーローの入場アニメーションが `animation: ... both` で指定され、最終状態で可視になっているか
- 違反が見つかったら、該当アニメーションを削除するか `:hover` / ヒーロー入場の `@keyframes` に置き換える

**レスポンシブチェック（必須）**
- `<style>` 内に `@media (max-width: 768px) { ... }` のブロックが存在する
- モバイル時、**すべてのグリッドが1列に折り返される**（`.grid-3`, `.grid-4`, `.grid-2`, `.hero-grid`, `.pricing-cards` 等）
- モバイル時、`.container` のパディングが `0 20px` に縮む
- モバイル時、見出し（h1/h2）が `clamp()` で縮小する or ブレークポイントで切り替わる
- モバイル時、CTAボタンが `width: 100%` になる
- 画像に `max-width: 100%; height: auto;` が効いている（縦横比維持）
- 比較テーブルや料金カードがモバイルで破綻していない（縦並びに組み替えるか横スクロール対応）

**全ての問題を自動修正してからユーザーに見せる。チェックリスト未達のまま提示は禁止。**

##### ③ ユーザーへの提示時に伝えること

- AI が仮で埋めた情報（仮のお客様の声・仮の数字・仮のFAQ等）を明示 → 「ここは後で実データに差し替えてください」
- AI が自動補完した画像を明示 → 「気に入らなければ Phase 5 で差し替え可能」
- 「スマホ表示も必ずチェックしてください」の案内

#### ユーザーとの修正ループ
1. ユーザーにブラウザで確認してもらう
   - **AI が自動補完した画像（ロゴSVG/Unsplash写真/pravatar顔写真）は、必ずこのタイミングで「画像差し替えたい箇所ありますか？」と一言添えて確認する**
   - **スマホ表示も必ず確認してもらう**（ブラウザの開発者ツール → デバイスモード → iPhone 14 等で切り替え）。崩れていればここで直す
2. フィードバックを受ける（**画像の差し替え・追加もこのタイミングで随時OK**）
3. `lp/index.html` を修正
   - ユーザーの画像への差し替え: 新しい画像を `lp/assets/` に保存し、HTMLの該当箇所の `src` を書き換える
   - 別のUnsplash写真に変えたい: AIが別のキーワードで再検索して URL を差し替える
   - 別の顔写真に変えたい: pravatar の seed を変えるか、ユーザー提供の顔写真に差し替え
   - ロゴを変えたい: ユーザー提供ロゴを `lp/assets/logo.png` に置いて差し替え
   - 画像を新しく追加したい: 指定のセクションに `<img src="./assets/[ファイル名]">` を追加
4. 再度ブラウザで確認してもらう
5. **「OK」「完成」「Figmaに送ろう」等の承認が出るまで繰り返す**

> AI が自動補完した画像はあくまで「仮入れ」。プレビューを見てから「やっぱり自社の写真を入れたい」「別の雰囲気の写真がいい」と思ったら遠慮なく言ってもらう。
> このフェーズは **完成度を上げるための主戦場**。画像も含めて納得いくまで何度でも変更する。

### Phase 6: Figma にインポート

**ユーザーがHTMLの出来に満足したら、AIが自動でFigma送り用HTMLをビルド → コードブロックで出力 → インポートを案内する。**

#### ステップ1: Figmaビルドの実行（AIが代行）

ユーザーの承認（「OK」「完成」「Figmaに送ろう」等）を得たら、AIはプロジェクトルートで以下を実行する:

```bash
npm run build:figma
```

このビルドが行うこと:
- `lp/index.html` を読む
- **CDN画像**（unDraw等）を fetch して Base64 に変換してHTMLに埋め込む
- **ローカル画像**(`./assets/*` など) もファイルから読み込んで Base64 に変換して埋め込む
- Phosphor Icons の Web Components `<script>` タグがあれば削除する
- `lp/figma-ready.html` として書き出す

→ これで html.to.design でも画像・アイコンが全て取り込まれる状態になる。

ビルドに失敗した画像があればログに警告が出る。失敗の原因は主に以下:
- CDN画像: URLが間違っている、またはネットワーク不通
- ローカル画像: ファイル名のタイプミスや、`lp/assets/` に配置し忘れ

警告が出たら原因を調べてユーザーに伝え、再ビルドする。

---

#### ステップ2: `lp/figma-ready.html` をユーザーがコピーしやすい形で出力

**ビルド完了後、AI は `lp/figma-ready.html` の中身をユーザーがそのままコピペできる形式で出力する。** ファイルパスだけ伝えるのは禁止（ユーザーはファイルを開く手間なく、チャット上で完結させたい）。

##### ① ファイルサイズを確認

```bash
wc -c lp/figma-ready.html
```

しきい値: **150KB（153600バイト）**

##### ② 150KB以下 → コードブロックで全文出力（基本ルート）

`lp/figma-ready.html` を Read ツールで読み込み、中身を ` ```html ` 〜 ` ``` ` のコードブロックにそのまま包んで出力する。

出力するメッセージの型:

```
LPが完成しました！以下のHTMLをコピーしてFigmaに貼り付けてください。

（ここに ```html から始まるコードブロックで figma-ready.html の中身を全量出力）

→ コードブロックの右上にある「Copy」ボタンでワンクリックコピーできます。
コピーできたら、次のステップ3のFigmaインポート手順に進んでください。
```

##### ③ 150KB超 → クリップボードに直接コピー（fallback）

Base64画像が大量に埋め込まれていてファイルが大きい場合、コードブロックに載せるとトークン制限を超える恐れがあるため、**AIがターミナルで直接クリップボードにコピーする**:

- Mac の場合: `cat lp/figma-ready.html | pbcopy`
- Windows の場合: `Get-Content lp/figma-ready.html | Set-Clipboard`

（AI が自分でコマンドを実行し、完了を確認する）

出力するメッセージの型:

```
LPが完成しました！
Figma送り用HTMLが [X.X] MB と大きいため、クリップボードに直接コピーしました。
（AI が `cat lp/figma-ready.html | pbcopy` を実行済み）

→ このまま Figma の html.to.design プラグインの Editor タブで Cmd+V（Windowsは Ctrl+V）で貼り付けるだけでOKです。
次のステップ3のFigmaインポート手順に進んでください。
```

---

#### ステップ3: Figmaインポート手順を案内

ユーザーに以下をそのまま伝える:

```
【Figma インポート手順】

1. Figmaを開く（ブラウザ or デスクトップアプリ）
2. 新しいファイルを作成
3. Cmd+K → 「html.to.design」を検索 → 実行
   （初回は「Try it out」をクリック）
4. 「Editor」タブを選択
5. 先ほどコピーしたHTMLを貼り付け（Cmd+V / Ctrl+V）
6. 「Create」をクリック
7. Import options:
   - Use Autolayout → ON
   - Create styles & variables → ON
   - HTML layer names → ON
   - 残りはOFFのまま
8. 「Proceed」をクリック

これでFigmaの編集可能なデザインになります。
画像もアイコンも全てFigmaにレイヤーとして取り込まれ、テキスト・色・配置を自由に変更できます。
```

> **重要**: コピーするのは `lp/index.html` ではなく **`lp/figma-ready.html`** の中身。`index.html` の方は外部CDN参照・ローカル相対パスのままでhtml.to.designから画像が取得できないため画像欠損が起きる。

## 注意事項

- ユーザーはプログラミング未経験者。HTML/CSSの説明は不要。「AIが作ったもの」として扱う
- コードを見せる必要はない。ブラウザプレビューで確認してもらう
- **全体を一気に生成**して見せる（完成イメージを早く見せた方がフィードバックしやすい）
- 修正は**何度でも**対応する
- 既存の資料（提案書、サービス概要等）が共有された場合は、内容を積極的にLPに反映する
- **毎回違うデザインにする**。前回と同じフォント・配色を使い回さない
